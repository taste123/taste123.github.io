Materi ini membahas tiga dokumen fundamental dalam eksekusi pengujian perangkat lunak: Test Scenario, Test Case, dan Bug Report. Ketiganya saling melengkapi untuk memastikan aplikasi bebas dari kesalahan.

## ğŸ—ºï¸ Test Scenario: "Apa yang Harus Diuji?"

**Test Scenario** adalah gambaran umum (*high-level*) tentang apa yang akan diuji. Ini menjawab pertanyaan "Fitur apa yang perlu kita validasi?".

Test scenario tidak menjelaskan *cara* menguji, melainkan hanya fokus pada *fungsionalitas* yang akan diuji.

### Template Test Scenario Sederhana

| Field | Keterangan |
| :--- | :--- |
| **ID Scenario** | Nomor unik untuk skenario |
| **Deskripsi** | Ringkasan skenario pengujian (fungsionalitas) |
| **Modul/Fitur** | Modul atau fitur yang sedang diuji |

---

## ğŸ§ª Test Case: "Bagaimana Cara Mengujinya?"

**Test Case** adalah dokumen yang jauh lebih rinci. Ini adalah langkah-langkah detail (*low-level*) yang menjelaskan *cara* melakukan pengujian untuk sebuah Test Scenario.

Setiap test case mencakup input spesifik, langkah-langkah eksekusi, dan hasil yang diharapkan (expected result).

### Template Test Case Sederhana

| Field | Keterangan |
| :--- | :--- |
| **ID Test Case** | Nomor unik untuk test case |
| **Deskripsi** | Ringkasan singkat tentang apa yang diuji test case ini |
| **Precondition** | Kondisi awal yang harus terpenuhi sebelum pengujian |
| **Test Steps** | Langkah-langkah detail untuk melakukan pengujian |
| **Test Data** | Data spesifik yang digunakan untuk pengujian (jika ada) |
| **Expected Result**| Hasil yang diharapkan sesuai dengan requirement |
| **Actual Result** | Hasil aktual yang muncul setelah pengujian |
| **Status** | Lulus/Gagal (Pass/Fail) |

## ğŸ› Bug Report: "Apa Masalahnya?"

**Bug Report** adalah laporan formal yang dibuat oleh tim QA ketika *Actual Result* tidak sesuai dengan *Expected Result*. Laporan ini adalah dokumen kunci untuk tim *developer* agar dapat mereproduksi dan memperbaiki masalah.

### Kunci Bug Report: Severity vs Priority

Saat melaporkan bug, penting untuk menentukan dua hal:

* **Bug Severity (Tingkat Keparahan):** Seberapa besar dampak bug ini terhadap *fungsionalitas* perangkat lunak.
    * **Critical:** Kegagalan total pada sistem, data rusak, aplikasi tidak dapat digunakan.
    * **Major (High):** Fungsionalitas utama tidak bekerja dengan benar.
    * **Minor (Medium):** Bug tidak memengaruhi fungsionalitas utama, namun menimbulkan ketidaknyamanan.
    * **Low:** Bug kosmetik (typo, warna salah) yang tidak merusak fungsi.

* **Bug Priority (Tingkat Prioritas):** Seberapa *mendesak* bug ini harus diperbaiki. Ini ditentukan oleh manajer produk/proyek.
    * **P1 - Urgent/Critical:** Harus diperbaiki *segera*.
    * **P2 - High:** Bug penting yang harus diperbaiki secepatnya.
    * **P3 - Medium:** Bug yang bisa diperbaiki di siklus rilis berikutnya.
    * **P4 - Low:** Bug minor yang bisa diperbaiki kapan saja jika ada waktu.

### Contoh Format Bug Report (untuk Aplikasi BMI)

| Field | Keterangan |
| :--- | :--- |
| **Bug Title** | Perhitungan BMI salah saat input berat 60kg dan tinggi 170cm |
| **Bug ID** | BMI-001 |
| **Step to Reproduce** | 1. Buka aplikasi BMI<br>2. Masukkan Berat = 60<br>3. Masukkan Tinggi = 170<br>4. Klik tombol "Hitung" |
| **Actual Result** | Hasil BMI = 12.5 |
| **Expected Result** | Hasil BMI seharusnya = 20.8 |
| **Severity** | **Major (High)** |
| **Priority** | **P2 - High** |
| **Assignee** | Developer |
| **Reporter** | SQA (Software Quality Assurance) |

## Cara Menghindari Bug

Cara terbaik menghindari bug adalah menerapkan praktik terbaik di seluruh siklus pengembangan:

* **Pahami Persyaratan:** Pastikan seluruh tim jelas mengenai apa yang harus dibuat.
* **Unit Testing:** Developer harus menguji kode mereka sendiri di tahap awal.
* **Code Review:** Minta pengembang lain meninjau kode untuk menemukan kesalahan logika.
* **Rencana Pengujian:** Buat *Test Plan* yang komprehensif.
* **Pengujian Otomatis:** Manfaatkan *automation testing* untuk deteksi bug yang lebih cepat dan berulang.
* **Kolaborasi Tim:** Tingkatkan komunikasi antara pengembang (developer) dan penguji (QA).
